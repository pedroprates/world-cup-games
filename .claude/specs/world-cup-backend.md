# World Cup 2026 Backend

## Summary

The app currently renders hardcoded, incomplete schedule data entirely on the client. This ticket adds a backend layer — Next.js 16 Route Handlers under `app/api/` — that serves the full 104-game fixture list (kickoff time, day, city, venue), all 48 team flags, and per-game broadcaster information (canonical name + self-hosted logo) to the frontend. The result is a single source of truth that the UI queries, making future fixture/broadcaster updates a backend-only concern.

## Investigation

**What exists:**
- `lib/fixtures.ts` — 14 hardcoded games (need 104 total for 48-team WC 2026)
- `lib/teams.ts` — 24 of 48 teams typed via `TeamCode` union; flags resolved via `flagcdn.com/{code}.svg`
- `lib/games.ts` — pure client-side `buildSchedule()` that imports fixtures/teams directly
- `lib/format.ts` — date/time formatters (Brasília timezone, `Intl.DateTimeFormat`)
- `app/page.tsx` — `"use client"` component, imports `buildSchedule` synchronously
- No API routes exist. No database. No broadcaster logos.

**What is missing:**
- 90 remaining fixtures (90 games) + all knockout fixtures
- 24 missing teams
- Broadcaster logos (5 files)
- All 4 API endpoints
- "GE TV" not yet in any fixture's channel list; "Cazé TV" spelling inconsistent

## UX Decisions

1. **Canonical broadcaster names**: Globo, SporTV, Cazé TV, Globoplay, GE TV. Drop "CazeTV" (no accent/space) and "ge.globo" variants.
2. **Broadcaster logos**: self-hosted in `/public/broadcasters/` as SVG (PNG fallback). Do not hotlink from broadcaster CDNs — URLs change and may block external refs.
3. **Backend architecture**: Next.js 16 Route Handlers in `app/api/` — no separate service. Data is read-only and low-traffic; a separate service adds operational overhead with no benefit.
4. **Game data**: curated static TypeScript (no scraping, no external API). FIFA has no public API; scraping is brittle. Group stage schedule is publicly known from the draw; knockout fixtures get added as results are known (follow-up ticket).
5. **Missing 24 teams**: all 48 qualified nations are known from the 2026 draw. Add all at once. `flagcdn.com` supports ISO 3166-1 alpha-2 codes for all nations; England (`gb-eng`) pattern is already handled.

## Technical Decisions

### Next.js 16 specifics
- Route Handlers live in `app/api/<segment>/route.ts`. Each file exports HTTP method functions (`GET`, etc.). `page.ts` and `route.ts` cannot co-exist at the same segment.
- **`params` is a Promise** in Next.js 15+ — dynamic handlers MUST `await ctx.params`. Use `RouteContext<'/api/games/[id]'>` (types generated by `next dev` / `next build` / `next typegen`).
- Route Handlers are NOT cached by default (changed in Next.js 15). Since data is fully static TypeScript, opt in with `export const dynamic = 'force-static'` on every read-only handler.
- Query params: use `request.nextUrl.searchParams` from `NextRequest` (import type from `next/server`).
- Return JSON via `Response.json(data)` (Web API standard; no `NextResponse.json` needed unless setting custom headers).

### Data model & API contracts

Add to `lib/types.ts`:
```ts
export interface Broadcaster {
  id: string;        // slug: "globo" | "sportv" | "caze-tv" | "globoplay" | "ge-tv"
  name: string;      // display: "Cazé TV"
  logo: string;      // path: "/broadcasters/caze-tv.svg"
}
```

Replace `Fixture.channels: string[]` with `Fixture.broadcasterIds: string[]` (referential to `Broadcaster.id`). Enrichment (joining teams + broadcasters) happens server-side in the API layer.

New `lib/api-types.ts` (response shapes shared by handlers and client):
```ts
export interface ApiGame {
  id: string;
  home: Team;
  away: Team;
  kickoff: string;          // ISO string — Date is not JSON-serializable
  city: string;
  venue: string;
  group: string;
  broadcasters: Broadcaster[];
}
export interface ApiGamesResponse      { games: ApiGame[] }
export interface ApiGameResponse       { game: ApiGame }
export interface ApiTeamsResponse      { teams: Team[] }
export interface ApiBroadcastersResponse { broadcasters: Broadcaster[] }
```

### Endpoints

| Method | Path                   | Query params                              | Response                    |
|--------|------------------------|-------------------------------------------|-----------------------------|
| GET    | `/api/games`           | `?group=Grupo+A` `?date=YYYY-MM-DD` `?team=BRA` | `ApiGamesResponse`  |
| GET    | `/api/games/[id]`      | —                                         | `ApiGameResponse` or 404    |
| GET    | `/api/teams`           | —                                         | `ApiTeamsResponse`          |
| GET    | `/api/broadcasters`    | —                                         | `ApiBroadcastersResponse`   |

- `?date` filters by Brasília day-key (matches `dayKey()` in `lib/format.ts`).
- `?team` filters by `TeamCode` on either home or away side.
- Invalid params → `400` with `{ error: string }`; unknown id → `404` with `{ error: "Game not found" }`.

### Code layout
- `lib/fixtures.ts` — raw 104 fixtures (data only, no logic)
- `lib/teams.ts` — raw 48 teams + `flagUrl()` (add missing 24 entries)
- `lib/broadcasters.ts` — **new**: 5 broadcasters with id, name, logo path
- `lib/api-types.ts` — **new**: API response shapes
- `lib/api/enrich.ts` — **new**: pure server-only helpers `enrichGame(fixture)` and `filterGames({group, date, team})`
- `lib/client/api.ts` — **new**: typed client wrappers (`fetchGames()`, `fetchGame(id)`, `fetchTeams()`, `fetchBroadcasters()`)
- `lib/schedule.ts` — **new**: refactored `buildSchedule` accepting `games: ApiGame[]` instead of importing FIXTURES directly

### Server vs Client
Keep `app/page.tsx` as `"use client"` (Pattern A — minimal churn). Replace direct `buildSchedule` import with a `useEffect` that calls `fetchGames()`, stores `ApiGame[]` in state, then passes games to the refactored `buildSchedule`. Countdown/live state stays client-side (needs `now` ticking every second).

### Implementer notes
- Do NOT put `route.ts` under `app/` root — `app/page.tsx` already owns `/`.
- Always `await ctx.params` in the `[id]` handler — breaking change since Next.js 15.
- `Date` is not JSON-serializable — keep `kickoff` as ISO string across the wire; convert to `Date` only inside `buildSchedule`.
- Locale-sensitive formatting (`lib/format.ts`) stays client-side; do not move into API response.
